# Financial AI Agent 技术文档

## 1. 系统概述

Financial AI Agent 是一个基于大语言模型的智能金融分析平台。系统采用前后端分离架构，后端以 FastAPI 为框架提供 RESTful API 与 SSE 流式接口，前端基于 Vue3 + TypeScript 构建交互界面。核心技术栈包括：LLM 对话服务、RAG 检索增强生成、多 Agent 协作编排、LLMOps 运维监控和 DataOps 数据管道。

## 2. 系统架构

### 2.1 分层架构

```
+--------------------------------------------------+
|                    前端展示层                       |
|         Vue3 + TypeScript + Pinia + ECharts       |
+--------------------------------------------------+
                        |
                   Nginx 反向代理
                        |
+--------------------------------------------------+
|                    API 网关层                      |
|              FastAPI + 中间件链                    |
+--------------------------------------------------+
        |          |          |           |
+----------+ +----------+ +--------+ +----------+
| Agent    | | RAG      | | LLM    | | 数据管道  |
| 编排层   | | 检索层   | | 服务层  | | 编排层   |
+----------+ +----------+ +--------+ +----------+
        |          |          |           |
+--------------------------------------------------+
|                   基础设施层                       |
|  PostgreSQL + Redis + ChromaDB + Prometheus       |
+--------------------------------------------------+
```

### 2.2 请求处理流程

一个典型的用户对话请求经过以下处理阶段：

1. **中间件链**：RequestIDMiddleware 注入追踪ID，SecurityHeadersMiddleware 添加安全响应头
2. **RAG 检索**：将用户查询向量化，在 ChromaDB 中检索相关知识片段，经 Cross-Encoder 重排序后取 Top-K
3. **意图分类**：编排器通过关键词匹配将用户查询分为 stock_analysis / stock_risk / full_analysis / investment_advice / general 等类型
4. **Agent 选择**：根据意图选择对应的 Agent 组合（投研 / 风控 / 策略）
5. **并行/串行执行**：多个 Agent 可并行执行以降低延迟，或串行执行以传递上下文
6. **结果聚合**：将各 Agent 输出按优先级合并为结构化分析报告
7. **LLMOps 记录**：记录本次调用的 Token 消耗、延迟、成本等指标

## 3. 核心模块设计

### 3.1 Agent 系统

#### 3.1.1 Agent 基类（BaseAgent）

所有 Agent 继承自 `BaseAgent`，核心接口包括：

- `execute(task, context, session_id)`：非流式执行，返回 `AgentExecutionResult`
- `execute_stream(task, context)`：流式执行，`AsyncIterator[str]`
- `_build_system_prompt()`：子类实现，构建系统提示词
- `_parse_output(raw_output)`：子类实现，解析 LLM 输出

Agent 配置通过 `AgentConfig` 数据类管理，包含名称、类型、描述、系统提示词、工具列表、最大迭代次数等参数。

#### 3.1.2 编排器（AgentOrchestrator）

编排器是多 Agent 系统的调度核心，负责：

- **意图分类**：基于中文关键词的规则匹配，支持 7 种意图类型
- **股票代码提取**：正则匹配 6 位数字股票代码
- **并行/串行调度**：并行模式下使用 `asyncio.gather` 并发执行，串行模式下按依赖顺序传递上下文
- **超时控制**：每个 Agent 有独立的 `asyncio.wait_for` 超时保护
- **会话记忆**：以 session_id 为键维护多轮对话上下文

#### 3.1.3 已有 Agent

| Agent | 类型 | 工具 | 职责 |
|-------|------|------|------|
| ResearcherAgent | researcher | 4个 | 基本面分析、财务解读、估值研判 |
| RiskAgent | risk | 2个 | 风险量化、合规审查、预警监控 |
| StrategistAgent | strategist | 0个 | 投资策略、组合建议、交易方案 |
| ReportAgent | report | 0个 | 结构化投研报告生成 |

### 3.2 RAG 系统

#### 3.2.1 知识库管理（KnowledgeBaseManager）

- 文档分块：使用 `RecursiveCharacterTextSplitter`，默认 chunk_size=500，overlap=50
- 向量存储：ChromaDB（开发环境），Milvus（生产环境），相似度算法使用 Cosine
- 批量写入：支持分批向量化与存储，batch_size 可配置

#### 3.2.2 混合检索器（HybridRetriever）

检索流程：

1. 用户查询向量化（Embedding）
2. ChromaDB 向量检索（n_results = top_k * 2）
3. Cosine 距离转换为相似度分数
4. 按 score_threshold 过滤低分结果
5. Cross-Encoder 重排序（可选，使用 ms-marco-MiniLM-L-6-v2 模型）
6. 查询扩展（可选）：通过 LLM 生成同义查询进行多路召回合并

#### 3.2.3 查询改写（QueryRewriter）

通过 LLM 对用户查询进行优化：口语化术语标准化（如"茅台" -> "贵州茅台(600519)"）、意图识别、同义词扩展。

### 3.3 LLMOps 监控系统

`LLMOpsMonitor` 以内存 + 线程安全的方式记录每次 LLM 调用：

- **调用日志**：model、tokens、latency、cost、success/error
- **每日统计**：按日期聚合的调用次数、Token 消耗、成本、成功率
- **滚动窗口**：内存中保留最近 10000 条记录
- **Prometheus 集成**：预留指标暴露接口
- **告警规则**：高错误率（>10%）、高延迟（P95 > 30s）、高成本（>$100/h）

### 3.4 DataOps 数据管道

`DataPipeline` 实现可组合的 ETL 框架：

- **PipelineStage**：每个阶段包含 extractor、transformer、validator、loader 四个可选的 callable
- **链式构建**：通过 `add_stage()` 方法链式添加阶段
- **同步/异步兼容**：`_safe_call()` 自动检测函数类型
- **预置抽取器**：Wind 行情数据、Tushare 财务数据（需配置后启用）

## 4. 数据库设计

### 4.1 核心表

| 表名 | 用途 | 关键字段 |
|------|------|----------|
| users | 用户账户 | username, email, hashed_password |
| chat_sessions | 对话会话 | session_id, user_id, agent_types |
| chat_messages | 消息记录 | session_id, role, content, agent_type |
| llm_call_logs | LLM调用日志 | model, tokens, latency_ms, cost_usd |
| knowledge_documents | 知识文档 | doc_id, title, category, indexed |
| document_chunks | 文档分块 | document_id, content, embedding_id |
| prompt_versions | Prompt版本 | version_id, agent_type, content |
| data_pipelines | 管道定义 | pipeline_name, schedule_cron |
| stock_cache | 数据缓存 | stock_code, data_type, data_json |

### 4.2 索引策略

- `llm_call_logs`：按 created_at 日索引 + model, created_at 复合索引
- `chat_sessions`：session_id, created_at 复合索引
- `document_chunks`：document_id, chunk_index 复合索引
- `stock_cache`：stock_code, data_type, updated_at 复合索引（缓存TTL管理）

### 4.3 TimescaleDB 优化

在 TimescaleDB 环境下，可将 `llm_call_logs` 的 `created_at` 字段创建为 hypertable，利用自动分区提升时序查询性能。

## 5. API 设计

### 5.1 对话接口

**POST /api/v1/chat**

请求体：
```json
{
  "session_id": "string",
  "message": "string (1-4000字符)",
  "agent_type": "researcher | risk | strategist | report (可选)",
  "stream": false,
  "enable_rag": true
}
```

**POST /api/v1/chat/stream**

同上，使用 SSE 协议逐步返回分析结果。响应格式：
```
data: {"content": "分析内容片段..."}
data: {"done": true, "session_id": "...", "tokens": 1234}
```

### 5.2 Agent 编排接口

- `POST /api/v1/agent/invoke`：直接调用 Agent 编排
- `GET /api/v1/agent/types`：获取可用 Agent 列表
- `GET /api/v1/agent/memory/{session_id}`：获取会话记忆
- `DELETE /api/v1/agent/memory/{session_id}`：清除会话记忆

### 5.3 RAG 接口

- `POST /api/v1/rag/query`：知识库检索
- `POST /api/v1/rag/knowledge/add`：添加知识文档
- `GET /api/v1/rag/knowledge/stats`：知识库统计

### 5.4 LLMOps 接口

- `GET /api/v1/llmops/metrics?days=7`：核心指标
- `GET /api/v1/llmops/daily?days=30`：每日趋势
- `GET /api/v1/llmops/recent?limit=50`：最近调用

### 5.5 数据接口

- `GET /api/v1/data/stock/{code}`：股票基本信息
- `GET /api/v1/data/stock/{code}/quote`：实时行情
- `POST /api/v1/data/pipeline/run`：触发数据管道

## 7. 安全设计

### 7.1 已实现的安全措施

- CORS 白名单控制（通过 CORS_ORIGINS 配置）
- 安全响应头（X-Content-Type-Options、X-Frame-Options、X-XSS-Protection、HSTS）
- 全局异常捕获避免敏感信息泄露
- 请求追踪 ID 用于审计
- 数据库连接池 pre_ping 防止连接失效

## 8. 扩展指南

### 8.1 添加新的 Agent

1. 在 `agents/` 目录新建 Python 文件
2. 继承 `BaseAgent`，实现 `_build_system_prompt` 和 `_parse_output`
3. 在 `models/schemas.py` 的 `AgentType` 枚举中添加新类型
4. 在 `orchestrator.py` 的 `_init_agents` 中注册
5. 更新 `INTENT_AGENT_MAP` 中的意图映射

### 8.2 接入新的数据源

1. 在 `dataops/pipeline.py` 中添加新的提取器函数
2. 创建对应的 `PipelineStage` 实例
3. 通过 `DataPipeline.add_stage()` 组合为完整管道

### 8.3 接入新的 LLM 模型

在 `core/llm.py` 的 `TokenCostCalculator` 中添加新的价格策略，模型配置通过环境变量 `LLM_MODEL` 和 `LLM_API_BASE` 动态指定，无需修改代码。

## 9. 已知限制与改进计划

### 当前限制

- Agent 工具函数为占位实现，需对接实际金融数据 API
- 意图分类基于简单关键词匹配，复杂查询可能分类不准确
- 前端数据看板展示静态示例数据，未对接实时行情
- 缺少用户认证体系的完整实现
- 流式对话暂为模拟实现，需接入 Agent 的真实流式能力

### 计划中的改进

- 引入结构化输出（JSON mode）提高 Agent 输出解析可靠性
- 使用向量化意图分类替代关键词匹配
- 增加 Agent 自我反思（Self-Reflection）能力
- 实现 Function Calling 的真实工具链
- 增加多用户并发场景的压力测试
